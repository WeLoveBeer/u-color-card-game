# U彩牌测试与质量保障文档

版本：1.0  
范围：规则引擎、AI、协议、服务端接口、客户端 UI、微信体验版和上线前验收

## 1. 目标

测试体系的目标不是只证明代码能运行，而是保证生成和手写代码都满足以下要求：

- 规则结果正确。
- 客户端不能绕过服务端裁决。
- AI 不提交非法操作，不读取隐藏信息。
- 协议字段和错误码在客户端、服务端保持一致。
- 随机结果可解释、可复盘。
- 好友房、断线重连、广告奖励等关键流程可在真机验证。

首版优先保障牌局正确性和公平性，其次保障体验完整性和性能稳定性。

## 2. 测试分层

### 2.1 静态检查

适用范围：

- TypeScript 类型。
- ESLint 规则。
- 协议类型。
- DTO 与错误码定义。

要求：

- 客户端和服务端都必须通过 TypeScript 类型检查。
- 协议消息类型、错误码、房间配置和游戏状态类型集中声明。
- 控制器、网关、场景脚本里不得散写协议字符串。
- 生成代码提交前必须通过静态检查。

建议命令：

```text
npm run typecheck
npm run lint
```

### 2.2 领域层单元测试

适用范围：

- `GameEngine`
- `RuleValidator`
- `EffectResolver`
- `TurnResolver`
- `DeckResolver`
- `ScoreResolver`
- 特殊牌 handler

要求：

- 领域层测试不依赖 NestJS、Redis、PostgreSQL、WebSocket、Cocos 或微信 SDK。
- 测试输入必须是明确的 `GameState`、`RuleConfig` 和玩家命令。
- 测试输出必须断言 `GameResult.ok`、新状态、领域事件和错误码。
- 每个新规则和新特殊牌必须补对应单元测试。

最低覆盖：

- 数字牌同色可出。
- 数字牌同数字可出。
- 不同色不同数字不可出。
- 玩家没有的牌不能打出。
- 非当前玩家不能操作。
- Skip 跳过下家。
- Reverse 在 2 人局等同 Skip。
- Reverse 在 3/4 人局改变方向。
- +2 默认让下家摸 2 并跳过。
- +2 叠加开启后可累计。
- +2 叠加关闭后拒绝叠加。
- +4 需要选择颜色。
- +4 质疑成功时出牌者摸 4 张。
- +4 质疑失败时质疑者摸 6 张。
- 变色牌更新当前颜色。
- 同色全出开启时允许自动带出同色牌。
- 同色全出关闭时拒绝同色全出动作。
- 气球牌随机结果由服务端生成。
- 忘喊 U 窗口正确打开、关闭和处罚。
- 出完最后一张牌立即结算。

### 2.3 协议契约测试

适用范围：

- HTTP DTO。
- WebSocket 客户端消息。
- WebSocket 服务端消息。
- 错误码。
- `DomainEvent` 到 `WsServerMessage` 的映射。

要求：

- 每个客户端请求都有参数合法和参数非法两类测试。
- 每个 WebSocket 消息都要验证 `seq` 回传、错误码和状态更新。
- `GameEventMapper` 必须固定多事件映射顺序。
- 服务端不得向其他玩家下发非公开手牌。
- 客户端 store 不得依赖服务端内部字段。

最低覆盖：

- 登录成功和 token 失效。
- 创建房间参数合法和非法。
- 加入不存在房间。
- 未轮到自己时出牌。
- 非法牌被服务端拒绝。
- 合法出牌返回状态和事件。
- 断线重连返回完整当前状态。
- 广告奖励超过限次返回错误。

### 2.4 服务端接口与对局流程测试

适用范围：

- NestJS HTTP API。
- WebSocket Gateway。
- Redis 房间状态。
- PostgreSQL 用户、对局记录、广告奖励和操作日志。

要求：

- 能通过接口测试完整跑完一局服务端牌局。
- 所有对局操作必须经过房间锁。
- 服务端状态写回 Redis 后，再广播给客户端。
- 对局结束后必须写入对局记录和必要操作日志。
- 测试环境和正式环境必须分开。

最低流程：

1. 用户登录。
2. 创建房间。
3. 多玩家加入房间。
4. 准备并开始。
5. 服务端洗牌、发牌。
6. 玩家出牌、摸牌、功能牌响应。
7. 触发结算。
8. 写入对局记录。
9. 清理或更新 Redis 状态。

### 2.5 随机对局模拟测试

适用范围：

- 规则引擎。
- AI 合法动作枚举。
- 托管快速路径。
- 特殊规则组合。

要求：

- 自动生成多种 `RuleConfig` 和牌局状态。
- 使用受控随机源运行大量对局。
- 任何一局都不能出现非法状态、死循环或无法结算。
- 失败用例必须输出 seed、规则配置和操作序列，方便复现。

建议检查：

- 所有玩家手牌、牌堆、弃牌堆中的卡牌 ID 不重复。
- 当前玩家必须属于玩家列表。
- 当前颜色不能是非法值。
- `pendingDrawCount` 和 `pendingDrawSource` 状态一致。
- 游戏结束后必须有 winner 和 rankings。
- AI 和托管永远只提交合法动作。

### 2.6 回放与复盘测试

适用范围：

- 对局记录。
- 操作日志。
- 随机种子 hash。
- 规则配置快照。

要求：

- 对局结束时保存规则配置快照、随机种子 hash、每一步操作、服务端校验结果和最终排名。
- 内部测试环境可以保存完整 seed 或初始牌序，用于问题复现。
- 正式环境不向普通用户暴露完整牌序和其他玩家手牌。
- 使用同一规则配置、初始牌序和操作序列重放时，最终结果必须一致。

用途：

- 排查争议对局。
- 复现线上 bug。
- 验证生成代码修改后没有改变历史规则结果。

### 2.7 客户端 UI 与交互测试

适用范围：

- Cocos 场景。
- 手牌交互。
- 动画队列。
- 断线重连表现。
- 规则弹窗和结算页。

要求：

- 客户端只展示服务端确认后的最终状态。
- 服务端确认前显示轻加载态并禁止重复提交。
- 服务端拒绝时卡牌回到原位并显示错误提示。
- 动画播放不改变领域状态。
- 客户端高亮可出牌只作为提示，最终合法性仍以服务端为准。

最低覆盖：

- 点击出牌。
- 拖拽出牌。
- 摸牌。
- 选择颜色。
- +4 响应。
- 喊 U。
- 抓忘喊。
- 断线后重连恢复。
- 结算页展示公平随机提示。

### 2.8 真机与微信体验版测试

适用范围：

- 微信登录。
- 微信分享。
- 好友房。
- WebSocket。
- 激励视频广告。
- 分包和远程资源。

要求：

- 至少两台手机通过微信分享进入同一房间并完整结算。
- 体验版可登录、分享、看广告领奖励。
- 合法域名、HTTPS、WSS、备案配置通过微信后台验证。
- 网络波动、切后台、锁屏和断线重连需要真机验证。

最低覆盖：

- Android 真机。
- iOS 真机。
- 弱网或网络切换。
- 首次启动。
- 非首次启动。
- 广告加载失败。
- WebSocket 断开后自动恢复。

## 3. AI 测试要求

AI 测试必须同时验证策略质量和规则边界。

必须覆盖：

- AI 能枚举所有合法动作。
- AI 不会提交非法出牌。
- AI 能直接获胜时一定优先获胜。
- AI 在下家剩 1 张时优先压制。
- AI 会选择自己最多且最有利的颜色。
- AI 会避免把回合送给只剩 1 张的玩家。
- 2 人局 Reverse 被当作 Skip 评估。
- 4 人局 Reverse 会改变方向判断。
- 同色全出开启时 AI 可打出同色全出专用牌，并正确评估自动带出的同色牌。
- 同色全出关闭时 AI 不会提交同色全出动作。
- +2 叠加开启时 AI 会选择叠加。
- +2 叠加关闭时 AI 不会非法叠加。
- +4 质疑开启时 AI 会根据记忆选择质疑或摸牌。
- 气球牌随机结果由服务端生成，AI 只提交出牌意图。
- 托管不会提交非法操作。
- AI 不读取隐藏牌堆和真实玩家手牌。

AI 评分可以调整，但以下行为不能被随机扰动推翻：

- 能直接获胜。
- 能阻止下家直接获胜。
- 不能提交非法动作。
- 不能读取隐藏信息。

## 4. 生成代码正确性要求

使用 AI 或代码生成工具生成代码时，必须按以下流程验收：

1. 明确生成目标对应的文档来源。
2. 生成后先跑类型检查和 lint。
3. 涉及规则、AI、协议、状态流转时必须补测试。
4. 涉及服务端接口时必须补接口测试或流程测试。
5. 涉及 UI 时必须真机或模拟器完成关键交互验证。
6. 修改规则结果时必须更新规则文档和测试用例。
7. 修改协议字段时必须同步客户端和服务端类型。

不能只靠肉眼检查生成代码。规则、协议和对局状态变更必须有自动化测试或可复现验收记录。

## 5. 推荐测试命令

首版建议预留以下命令：

```text
npm run lint
npm run typecheck
npm run test
npm run test:domain
npm run test:api
npm run test:simulate
npm run test:replay
```

如果客户端和服务端分目录维护，可以拆成：

```text
npm run test:client
npm run test:server
```

## 6. CI 门禁建议

提交到主分支前至少通过：

- lint。
- typecheck。
- 领域层单元测试。
- 协议契约测试。
- 服务端核心接口测试。

上线前额外通过：

- 随机对局模拟测试。
- 回放一致性测试。
- 两台真机好友房完整结算。
- 微信体验版登录、分享和广告验证。
- 包体、分包和资源加载检查。

## 7. 验收标准

M1 人机原型：

- 不连接服务端也能完整打一局人机。
- 本地规则包通过领域层单元测试。
- AI 不提交非法动作。

M2 服务端规则引擎：

- 服务端可完整跑完一局牌局。
- 规则引擎核心用例全部通过。
- 非法出牌、非当前玩家操作、缺少颜色选择等错误码正确返回。

M3 好友房：

- 两台手机可通过微信分享进入同一房间并完整结算。
- 断线 15 秒后进入托管，重连后恢复最新状态。
- 并发操作不会覆盖房间状态。

M4 微信能力与广告：

- 体验版可登录、分享、看广告领奖励。
- 广告奖励由服务端限次。
- 广告不影响好友房公平胜负。

M5 上线前打磨：

- 日志监控可用于定位对局问题。
- 随机种子摘要和公平提示正常展示。
- 资源加载、分包和弱网表现通过真机验证。
- 发现问题可以通过日志和操作序列复现。
